# Peer-Level Identity for tidepool — Design Spec

**Date:** 2026-04-14
**Status:** Approved for planning

## Problem

Today, `tidepool register <name>` generates a fresh self-signed cert and private key at `$TIDEPOOL_HOME/agents/<name>/identity.{crt,key}` for every agent. The intent reads as "one identity per agent." But `server.ts:buildTlsOptions` only loads the **first-registered agent's** cert as the TLS server identity, and the outbound proxy at `server.ts:521-524` reuses the same first-agent cert for every mTLS egress. All other agents' certs are written to disk and never touched at the transport layer.

This produces three problems:

1. **Conceptual confusion.** Users are told in the README to pair `register`'d agents 1:1 with Claude sessions. That implies each agent is a peer. It isn't: the host-level identity is "whichever agent you happened to register first."
2. **Ceremony for multi-agent setups.** To run two local Claude Code sessions as addressable peers, docs previously said to stand up two `TIDEPOOL_HOME`s, two `tidepool serve` processes, two fingerprints to exchange. The right answer for "multi-agent on one box" is one tidepool with two tenant agents behind it.
3. **Local agent-to-agent is undiscovered.** The local proxy app (`server.ts:466-508`) already forwards `POST /:tenant/:action` to `config.agents[tenant].localEndpoint` when the tenant isn't a remote. So Agent A can already send to Agent B on the same host. The docs don't say this and the setup instructions actively work against it.

## Goal

Make "real multi-agent on one box" the happy path. One tidepool host = one peer identity. Agents are tenant routes behind that peer. Local agent-to-agent is a documented first-class pattern with no extra config.

## Non-goals

- **No change to the A2A wire protocol.** Message envelopes, SSE framing, agent cards, all unchanged.
- **No auth between co-hosted agents.** Agents on one tidepool (same home, same user, same process) trust each other. If a user wants cross-agent isolation, they can run separate homes — the old topology still works.
- **No SNI / multi-cert TLS.** We're collapsing to one peer cert, not adding per-agent TLS identity.
- **No migration tooling.** Breaking change in v0.0.x. Users re-run `init` and re-exchange fingerprints with friends.

## Model

**Peer identity** — the host. One cert at `$TIDEPOOL_HOME/identity.{crt,key}`. Generated by `init`. Its fingerprint is what friends pin.

**Agent** — a named tenant route on a peer. Declared in `[agents.<name>]` in `server.toml`. Has a `localEndpoint` where its HTTP app (typically the a2a-claude-code-adapter) listens.

**Friend** — a peer fingerprint + handle. Unchanged on the wire. `agents = [...]` scope still restricts which local agents an inbound friend can reach.

**Remote** — a `[remote.<handle>]` entry mapping a local handle to a remote peer's endpoint + remote tenant + remote peer fingerprint. Unchanged.

## Routing rule

At the local HTTP port, `POST /:handle/:action` resolves `:handle` in this order (this matches existing behavior — no code change to the handler, we're pinning the behavior down as a rule):

1. Registered `[remote.<handle>]` → mTLS egress with the **peer cert**, pinning the remote peer's fingerprint.
2. Registered `[agents.<handle>]` → direct `fetch()` to `agent.localEndpoint`, no TLS.
3. Neither → 404.

## What changes

### 1. `init` generates the peer identity

`runInit(configDir)` also calls `generateIdentity({ name: "peer", certPath: "<configDir>/identity.crt", keyPath: "<configDir>/identity.key" })` if the files don't already exist. Prints the peer fingerprint.

### 2. `register` stops generating certs

`runRegister` only edits `server.toml` to add the `[agents.<name>]` entry. No cert/key writes. No `agents/<name>/` directory. Its return value changes from `{ fingerprint }` (the agent cert) to `{ peerFingerprint }` (read from the peer cert) so CLI output still prints a useful fingerprint.

### 3. TLS server uses the peer cert

`buildTlsOptions(configDir)` reads `<configDir>/identity.{crt,key}` instead of `<configDir>/agents/<firstAgent>/identity.{crt,key}`. The `firstAgent` variable is deleted.

### 4. Outbound mTLS uses the peer cert

The local app's remote-egress branch (`server.ts:521-524`) reads `<configDir>/identity.{crt,key}` instead of the first agent's. The `buildPinnedDispatcher` call is unchanged.

### 5. `whoami` prints the peer

`runWhoami(configDir)` returns `{ peerFingerprint, agents: string[] }`. CLI output shows one fingerprint and a list of agent names. No per-agent fingerprints.

### 6. Fixtures + README

- `fixtures/server.toml` gets a leading comment: "Agents are tenant routes; the peer cert lives at $TIDEPOOL_HOME/identity.crt."
- `packages/a2a-claude-code-adapter/README.md` drops any "two TIDEPOOL_HOMEs" language and adds a section: "Running two Claude sessions on one host" with the one-home / two-agents recipe.

## Data flow — two Claudes, one home

```
Terminal A (alice session)              Terminal B (bob session)
  a2a-claude-code-adapter                 a2a-claude-code-adapter
    listens at :18800                       listens at :18801
    --agent alice                           --agent bob
         ^                                       ^
         | local HTTP                            | local HTTP
         |                                       |
     +---+---------------------------------------+---+
     |           tidepool serve                  |
     |  public  mTLS   :9900 (peer cert)             |
     |  local   HTTP   :9901                         |
     |  server.toml:                                 |
     |    [agents.alice]  localEndpoint = :18800     |
     |    [agents.bob]    localEndpoint = :18801     |
     +-----------------------------------------------+
```

Alice's session sends `POST http://127.0.0.1:9901/bob/message:send` → local app resolves `bob` to `[agents.bob]` → forwards to `http://127.0.0.1:18801/message:send` → bob's adapter surfaces `<channel source="a2a" task_id="...">` in Claude session B → Claude B calls `a2a_reply` → response flows back over the same open HTTP connection.

## Failure modes

- **Peer cert missing when `serve` starts.** Fatal, with a clear message pointing at `tidepool init`.
- **`register` called before `init`.** Fatal, same guidance.
- **Agent handle collides with a remote handle.** Validate at config load: reject if `Object.keys(config.agents)` intersects `Object.keys(remotesConfig.remotes)`.
- **Agent's `localEndpoint` is down.** Existing 504 path in `server.ts:504`. Unchanged.

## Testing

- Unit: `runInit` writes `identity.{crt,key}` with 0o600 on the key.
- Unit: `runRegister` does NOT write any cert files; throws if peer identity missing.
- Unit: `buildTlsOptions` loads `<dir>/identity.{crt,key}` and ignores `agents/*`.
- Unit: `runWhoami` returns peer fingerprint + agents list.
- Integration: two agents on one tidepool, agent A → agent B round-trip over `/:handle/message:send` (no mTLS hop, verified by checking that the handler takes the local-passthrough branch).
- E2E: existing mTLS pinning / handshake / rate-limit tests still pass after the cert source change. They already use single-agent peers, so the peer-cert switchover should be transparent.

## Risk

Low surgery. The local-passthrough code path is already implemented and tested (see `server.ts:466-508`). The mTLS egress branch is functionally unchanged; only the cert source moves. The biggest risk is test fixtures that assume `agents/<name>/identity.crt` exists — each test suite's fixture setup needs an `identity.{crt,key}` at the config root instead.
